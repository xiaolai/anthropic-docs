# Anthropic API & SDK — Known Issues

> Daily issue-tracker scans of
> [`anthropics/anthropic-sdk-typescript`](https://github.com/anthropics/anthropic-sdk-typescript)
> (the bug-tracker repo for this skill — see `config.json.upstream.bugTrackerRepo`)
> land confirmed user-impacting bugs here as `### KI N — <title>` entries.

## How entries land here

The research agent's Part B reads new bug-labeled issues in the upstream
repo. For each:

- **`added_known_issue`** → a new `### KI N — <title>` section is added
  here with symptom / reproduction / workaround / link.
- **`added_rule`** → if the bug is auto-correctable at edit time, the
  fix becomes a new rule in [`rules/messages-api.md`](rules/messages-api.md)
  instead of an entry here.
- **`skipped`** → recorded in `state.json.researchedIssues` with a
  reason; not added here (typically feature requests, environment-
  specific, or already documented).

### KI 1 — Opus 4.6 / Sonnet 4.6: regression in parallel tool calls via Batch API

**Issue:** [anthropics/anthropic-sdk-typescript#956](https://github.com/anthropics/anthropic-sdk-typescript/issues/956)
**Status:** Open (as of 2026-05-18). Acknowledged by Anthropic; scale-dependent reproduction confirmed.

**Symptom:** When using the Message Batches API with 40+ tool definitions, `claude-opus-4-6` and `claude-sonnet-4-6` produce only **1 tool call** per batch response, regardless of `tool_choice: "any"` or `max_tokens`. The equivalent request on `claude-opus-4-5-20251101` produces 39+ parallel tool calls.

**Trigger:** Large tool count (≥ ~40 tools) in a single Batch API request with 4.6-series models.

**Workaround:** Use `claude-opus-4-5-20251101` or `claude-sonnet-4-5-20251001` for workflows requiring parallel tool calls in Batch API. If 4.6 is required, break large tool sets into multiple serial batch requests.

### KI 2 — `toolRunner` + `MessageStream` do not propagate `container.id` across iterations

**Issue:** [anthropics/anthropic-sdk-typescript#964](https://github.com/anthropics/anthropic-sdk-typescript/issues/964)
**Status:** Open (labeled `bug`; fixes pending in PRs #966, #974, #981, #994 as of 2026-05-18).

**Symptom:** When using `client.beta.messages.toolRunner` (or `messages.stream()` + `finalMessage()` directly) with `code_execution_20260120` plus client-side tools, the container ID returned in iteration N is **not forwarded** to iteration N+1. This produces the error: `container_id is required when there are pending tool uses generated by code execution with tools`.

Attempting to work around by calling `runner.setMessagesParams({ container: id })` causes the model to enter an **infinite duplicate tool-calling loop**.

**Root cause:** `MessageStream.#accumulateMessage()` does not carry over the `container` field from the `message_delta` event. Affects both `toolRunner` and raw `MessageStream` callers.

**Workaround:** Implement the multi-turn loop manually using `client.beta.messages.create` directly. On each turn, read `response.container.id` and pass it as `container` in the next request params.

### KI 3 — SDK strips directory paths from filenames in multipart uploads (breaks `skills.versions.create`)

**Issue:** [anthropics/anthropic-sdk-typescript#968](https://github.com/anthropics/anthropic-sdk-typescript/issues/968)
**Status:** Open (labeled `spec-change`; fix pending as of 2026-05-18).

**Symptom:** The SDK's multipart form-data helper strips directory paths from filenames (`my-skill/SKILL.md` → `SKILL.md`) before upload. `client.beta.skills.versions.create()` requires the top-level directory in the filename; the API returns: `400 SKILL.md file must be exactly in the top-level folder.`

**Trigger:** Calling `client.beta.skills.versions.create()` with a `SKILL.md` file path that includes a directory prefix.

**Workaround:** Build the `FormData` manually, setting the filename explicitly:
```typescript
const form = new FormData();
form.append('files', blob, 'my-skill/SKILL.md');
await fetch('https://api.anthropic.com/v1/beta/skills/versions', { method: 'POST', body: form, headers: { 'x-api-key': key, 'anthropic-version': '2023-06-01', 'anthropic-beta': 'skills-2025-10-02' } });
```

### KI 4 — `toBase64` stack overflow on inputs > ~100 KB in non-Node environments

**Issue:** [anthropics/anthropic-sdk-typescript#932](https://github.com/anthropics/anthropic-sdk-typescript/issues/932)
**Status:** Open (labeled `code-change`; fix pending as of 2026-05-18).

**Symptom:** In browser or Cloudflare Workers environments (no Node.js `Buffer`), calling any SDK method that base64-encodes a large payload (e.g. image upload > ~100 KB) throws `RangeError: Maximum call stack size exceeded`. The `toBase64` function in `src/internal/utils/base64.ts` uses a call-stack-based approach that overflows on large `Uint8Array` inputs.

**Trigger:** Image or file payloads > ~100 KB in browser / edge runtime without Node.js `Buffer`.

**Workaround:** Pre-encode the data to a base64 string before passing to the SDK:
```typescript
// Browser workaround
const reader = new FileReader();
reader.readAsDataURL(file); // gives "data:<type>;base64,<encoded>"
const base64 = reader.result.split(',')[1];
```
Or use `btoa` with chunked string conversion for `Uint8Array`.

### KI 5 — `toolRunner` drops `defaultHeaders` on follow-up requests

**Issue:** [anthropics/anthropic-sdk-typescript#922](https://github.com/anthropics/anthropic-sdk-typescript/issues/922)
**Status:** Open (labeled `code-change`; 5 comments confirming, as of 2026-05-18).

**Symptom:** `client.messages.toolRunner` (and `client.beta.messages.toolRunner`) correctly sends `defaultHeaders` on the first API call. Subsequent calls in the tool-use loop (sending `tool_result` back) **do not include** those headers. This breaks proxy / gateway setups that rely on custom auth headers (e.g. Cloudflare AI Gateway `cf-aig-authorization`). The second request returns `401 authentication_error`.

**Trigger:** Any `toolRunner` workflow with `defaultHeaders` set on the client, especially when routing through a proxy.

**Workaround:** Implement the multi-turn loop manually using `client.messages.create` directly instead of `toolRunner`. Each `create` call will correctly include `defaultHeaders`.
