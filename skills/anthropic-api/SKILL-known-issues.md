# Anthropic API & SDK — Known Issues

> Daily issue-tracker scans of
> [`anthropics/anthropic-sdk-typescript`](https://github.com/anthropics/anthropic-sdk-typescript)
> (the bug-tracker repo for this skill — see `config.json.upstream.bugTrackerRepo`)
> land confirmed user-impacting bugs here as `### KI N — <title>` entries.

## How entries land here

The research agent's Part B reads new bug-labeled issues in the upstream
repo. For each:

- **`added_known_issue`** → a new `### KI N — <title>` section is added
  here with symptom / reproduction / workaround / link.
- **`added_rule`** → if the bug is auto-correctable at edit time, the
  fix becomes a new rule in [`rules/messages-api.md`](rules/messages-api.md)
  instead of an entry here.
- **`skipped`** → recorded in `state.json.researchedIssues` with a
  reason; not added here (typically feature requests, environment-
  specific, or already documented).

### KI 964 — `toolRunner` drops `container.id` across iterations; `messages.stream()` also affected

**Status:** Open (as of 2026-05-20). PRs #966, #974, #981, #994 partially fix `BetaToolRunner` but not the lower-level `MessageStream` issue.

**Symptom:**  
When using `client.beta.messages.toolRunner` (or `client.beta.messages.stream` + `.finalMessage()`) with `code_execution_20260120` alongside client-side tools, the response's `container.id` is not forwarded to the next iteration. This produces:

```
container_id is required when there are pending tool uses generated by code execution with tools.
```

Using `runner.setMessagesParams({ container: { id: ... } })` as a workaround causes an infinite duplicate tool-call loop because it drops the current assistant/tool turn from the messages history.

**Root cause:**  
`MessageStream.#accumulateMessage()` (`src/lib/MessageStream.ts`, ~L576) processes `message_delta` events and updates `stop_reason`, `stop_sequence`, and `usage` — but never copies `event.delta.container` to the accumulated snapshot, even though `RawMessageDeltaEvent.Delta` declares `container: Container | null`.

**Reproduction:**
```typescript
const runner = client.beta.messages.toolRunner({
  model: "claude-opus-4-6",
  tools: [
    { ...myTool, allowed_callers: ["code_execution_20260120"] },
    { type: "code_execution_20260120", name: "code_execution" },
  ],
  messages: [{ role: "user", content: "..." }],
  stream: true,
});

for await (const messageStream of runner) {
  for await (const event of messageStream) {}
  const msg = await messageStream.finalMessage();
  // msg.container is null even when container was active
}
```

**Workaround:**  
None fully satisfactory while the SDK bug is open. Options:
- Use `client.beta.messages.create` (non-streaming) for code-execution flows where you need `container.id` — the container is returned correctly on the non-streaming path.
- Or manually track container ID via streaming events (look for `container` in `message_start` events) instead of relying on `finalMessage().container`.

**Link:** [anthropics/anthropic-sdk-typescript#964](https://github.com/anthropics/anthropic-sdk-typescript/issues/964)
